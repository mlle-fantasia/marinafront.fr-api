Résumé de l'énoncé

Ce projet consiste à créer un jeu de plateau en JavaScript dans lequel 2 joueurs évoluent chacun leur tour pour s’affronter. Il faut tout d’abord générer un plateau de jeu avec des obstacles et des armes positionnées aléatoirement. Tour par tour les joueurs vont se déplacer de maximum 3 cases horizontalement ou verticalement. S’il arrive sur une arme, ils s’en emparent et s’ils arrive proche de l’autre joueur, ils doivent s’affronter. Il doit ne rester qu’un gagnant.

L’architecture orienté objet

Le projet imposait d’utiliser une architecture orienté objet. Cette méthode permet principalement de structurer le code en plusieurs fichiers et le rendre plus lisible. L’un des principes fondamental de la programmation orientée objet est l’encapsulation. Chaque objet possède des attributs et des méthodes et une fois qu’un objet est créé, on peut faire appel aux méthodes de l’objet. Un autre principe est l’héritage. Un objet peut hériter d’un autre objet et pourra profiter des attributs et des méthodes de la classe parente. Cela permet de ne pas répéter le code.

Les pages

Le projet se découpe en 3 pages, - la page d’accueil où l’on va choisir l’univers du jeu et les personnages - la page de grille du jeu où les joueur pourront se déplacer tour par tour - la page de combat où un des deux combatant va perdre la partie Pour communiquer entre les pages, j’ai utilisé le localstorage. Par exemple sur la première page, l’orsqu’on choisi l’univers je l’ajoute dans le localstorage et ensuite en arrivant sur la deuxième page, le programme va récupérer l’information pour pouvoir afficher la bonne image de fond d’écran et les bonnes couleurs sur la grille.

La grille

La grille est un simple tableau de cellules, un tableau à double entrées. D’une part je crée le tableau avec des objets de la classe “cellule” où chaque objet cellule possède des attributs comme une position x, une position y, un tableau d’objet vide ou remplis s’il y a des objets (joueur, obstacle ou arme) placés sur cette case... Ensuite, pour afficher la grille dans le navigateur, je tourne dans ce tableau et je crée des div et s’il y a des objets a placer sur la case, j’ajoute des classes css pour changer l’affichage. par exemple, les obstacles on un background-color différent et les joueurs ou les accessoires ont un background-image particulier.

Test de déplacement

Les joueurs doivent pouvoir se déplacer de 0 à 3 case horizontalement et verticalement. A chaque tour il faut donc calculer sur quelles cases il peut se déplacer par rapport à sa position et en fonction des obstacle sur sa route. car il ne peut pas se déplacer sur une case ou se trouve un obstacle ou un joueur et il ne peut pas non plus passer par dessus. Le code suivant est un extrait de ce test de déplacement.

static showMove(grid, x, y, ptfort) {
let tabMovableCell = [];
let avantage = false;
if (ptfort.value === “fast”) {
avantage = this.calculChanceAvantage(ptfort);
}
this.testMoveLeft(x, y, grid, avantage, tabMovableCell);
this.testMoveTop(x, y, grid, avantage, tabMovableCell);
this.testMoveBottom(x, y, grid, avantage, tabMovableCell);
this.testMoveRight(x, y, grid, avantage, tabMovableCell);
return tabMovableCell;
}

static testMoveLeft(x, y, grid, avantage, tabMovableCell) {
let nbCaseMaxToMove = 3;
if (avantage) {
nbCaseMaxToMove = 4;
}
for (let i = y - 1; i >= y - nbCaseMaxToMove; i--) {
if (this.isExistCell(x, i)) {
let cellEmpty = this.isFreeCell(x, i, grid);
if (cellEmpty) {
grid[x][i].movable = true;
tabMovableCell.push({ x: x, y: i });
} else {
if (
grid[x][i].objects[0] instanceof Player ||
grid[x][i].objects[0] instanceof Obstacle ||
(grid[x][i].objects[1] && grid[x][i].objects[1] instanceof Player)
) {
grid[x][i].movable = false;
return;
} else {
grid[x][i].movable = true;
tabMovableCell.push({ x: x, y: i });
}
}
}
}
}

Ici, la fonction showMove() reçoit en paramètre la grille et la position du joueur. Ensuite on va tester le mouvement dans chaque direction. Par exemple, pour les mouvements vers la gauche, on fait une boucle où au départ i est égale à la postion de joueur et tant que i est plus grand ou égale à la position du joueur moins 3 cases. Dans chaque boucle on vérifie tout d’abord si la case existe (si elle n’est pas en dehord de la grille). ensuite, on vérifie si la case est libre. Si la case n’est pas libre, on vérifie quel est l’objet sur cette case. Si l’on peut se déplacer dessus : - Je change l’attribut “movable” de la cellule. Cet attribut permet d’une part d’afficher la case avec une couleur différente ou non et d’autre part de savoir si le joueur doit se déplacer l’orsque l’utilisateur clique sur une case. - J’ajoute ses coordonées dans un tableau. Ce tableau me permet à la fin du tour de réinitialiser les cellules en évitant de boucler sur la grille entière.

Corriger les bugs

Le bug numéro 2 est un conflit éventuel entre deux IDs identiques.
Voici comment était générer l'id :

var newId = "";
var charset = "0123456789";
for (var i = 0; i < 6; i++){
newId += charsett.charAt(Math.floor(Math.random() * charset.length));
}
Lors de la création d’une todo, pour générer l’id, le programme va piocher 6 fois un chiffre au hasard dans la chaîne de caractère charset = “0123456789” Résultat : créer deux ids identiques est difficile (1 chance sur 1 million) mais pas impossible.
il existe plusieurs possibilités pour créer des id uniques :
- en reprenant le code existant et en déplaçant la génération de l’id dans une fonction récursive qui va boucler dans les todos et vérifier si l’id qui vient d’être généré n’existe pas déja, si oui la fonction et rappelée pour regénérer un id, puis puis la vérification est effectuée à nouveau et ceci jusqu’a ce que l’id n’existe pas déjà.
- soit on utilise un uuid, c’est une chaîne de caractères générée aléatoirement de 128 bit. Bien que la probabilité qu’un UUID soit dupliqué n’est pas nulle, elle est suffisamment proche de zéro pour être négligeable.
- soit en reprenant l’idée d’id très long où, le risque de duplication est quasi null, on ne change pas le système existant, mais on augment le nombre de boucle, au lieu de 6 on peux piocher 36 chiffres qui définiront l’id. Ainsi, le risque d’avoir un id identique sera fortement diminué.
- soit on utilise un nombre incrémenté à chaque nouvelle todo cad que même si on supprime la todo, la todo suivante aura un id incrémenté.
- soit on utilise un timestamp. Celui-ci correspond au nombre de seconde qui s’est écoulé depuis le 1er janvier 1970, Ainsi nous sommmes assurés de son unicité dans la mesure où ce projet de todolist ne peut pas demander deux ids en même temps. (solution utilisée)
